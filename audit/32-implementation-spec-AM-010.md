# AirMentor Implementation Spec AM-010

## Problem statement
The repository still contains dormant mock flows, tracked prototype surfaces, and tracked non-runtime artifacts that no longer represent the live AirMentor product surface. Around that committed drift, local scratch probes and ignored generated artifacts also add noise during development. Together they increase search noise, review ambiguity, and onboarding cost.

## Exact code locations
- Dormant mock path:
  - `src/system-admin-mock-app.tsx`
  - `src/system-admin-mock-data.ts`
  - `tests/system-admin-mock-data.test.ts`
  - `scripts/system-admin-mock-acceptance.mjs`
- Prototype and local scratch artifacts:
  - `caludeui.jsx`
  - `chatgpt.jsx`
  - `tmp_eval_progress.cjs`
  - `tmp_eval_progress_32889.cjs`
  - `tmp_pg_activity_32889.cjs`
- Tracked non-runtime artifact:
  - `Pre_Project_Presentation_Template_8th Sem 1.pdf`
- Ignored generated artifact:
  - `air-mentor-api/scripts/__pycache__/curriculum_linkage_nlp.cpython-313.pyc`
- Transitional tooling with live coupling:
  - `air-mentor-api/scripts/generate-academic-parity-seed.ts`
  - `src/data.ts`
  - `src/domain.ts`

## Root cause
The product evolved from mock and prototype stages into a live backend-backed system, but the repo never got a hard cleanup boundary between active runtime, intentionally retained transitional tooling, dead committed artifacts, and local scratch space.

## Full dependency graph
- dormant mock app -> mock data -> mock tests and mock Playwright acceptance
- transitional parity tooling -> frontend local model files -> backend seed generation
- committed prototype/non-runtime artifacts -> developer search results, code review noise, ownership ambiguity
- local scratch probes and ignored generated artifacts -> working-tree noise during debugging and audit work

## Affected user journeys
- engineering onboarding
- bug triage and code search
- test selection and maintenance
- audit and architecture review accuracy

## Risk if left unfixed
- engineers keep opening or modifying non-authoritative surfaces
- review time remains inflated by dead or transitional code
- accidental dependencies on dormant artifacts persist
- repo narrative stays less trustworthy than the live product

## Target future architecture or behavior
- active runtime, retained transitional tooling, archived experiments, and deleted artifacts are clearly separated
- the repo root only contains live or deliberately maintained surfaces
- any intentionally retained experimental code is labeled and isolated

## Concrete refactor or fix plan
1. Classify each non-runtime artifact as:
   - keep active
   - move to archive
   - delete
2. Remove clearly dead committed files:
   - zero-byte placeholders
   - non-runtime presentation artifacts if not required
3. Relocate or delete local scratch probes and ignored generated bytecode from the active working surface.
4. Archive or remove the dormant mock admin path unless it still serves a named acceptance purpose.
5. Decide whether academic parity seeding should remain transitional tooling or be replaced with backend-native fixtures.
6. Add repo hygiene checks for temp files and generated artifacts.

## Sequencing plan
- Do after AM-013 so verification is stable.
- Separate committed cleanup from local scratch cleanup and from any parity-tool replacement work.
- Land mock-path cleanup only after confirming no active script or reviewer depends on it.

## Migration strategy
- Move uncertain artifacts into an explicit archive path first instead of deleting immediately.
- Update tests and scripts in the same change as any file removal.
- Keep parity seed tooling until replacement fixtures exist.

## Testing plan
- Run any remaining mock-related tests or scripts before deletion.
- Run standard frontend/backend build and test baselines after cleanup.
- Validate that search-based developer workflows and docs no longer point at removed files.

## Rollout plan
- Land in small cleanup PRs grouped by artifact family:
  - root temp/prototype cleanup
  - mock admin cleanup
  - parity-tool boundary cleanup

## Fallback / rollback plan
- Archived artifacts can be restored if a hidden dependency appears.
- Avoid deleting parity tooling until replacement coverage is verified.

## Acceptance criteria
- Repo root no longer contains dead committed placeholders or ambiguous prototype surfaces outside an explicit archive boundary.
- Any retained prototype or mock surface is explicitly archived or labeled as non-runtime.
- Local scratch probes and ignored generated bytecode are cleared from the active working surface.
- No active script, test, or build step depends on deleted artifacts.
- The repo coverage ledger cleanly separates active runtime from archival surfaces.

## Open questions
- Does the team want to retain the mock admin path for demos, or is it fully obsolete?
- Should `generate-academic-parity-seed.ts` be replaced as part of cleanup or tracked as a separate migration?
