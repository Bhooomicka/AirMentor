# AirMentor Feature Inventory And Traceability Matrix

## What this area does
This is the master end-to-end feature map for AirMentor. It ties product features to users, frontend entry points, backend endpoints, core files, database tables, test coverage, issue IDs, current risk, and lifecycle state.

## Confirmed observations
- AirMentor is not one homogeneous product. It is a set of connected academic, admin, proof, and tooling features with different maturity levels.
- Several features are fully live but still carry transitional dependencies in adjacent code.
- Feature tracing required joining frontend routes, backend endpoints, database tables, and test files rather than reading one layer in isolation.
- No student-authenticated runtime was found in the current code. “Student shell” and “risk explorer” are staff-operated proof tools for course leaders, mentors, HoDs, and system admins, not student self-service features.
- The current repo also includes engineering/operator surfaces that materially affect runtime trust:
  - startup diagnostics
  - structured telemetry
  - CI verification and scheduled proof/browser cadence

## Current-state reconciliation (2026-03-28)
- This matrix now needs to be read with two current-state realities in mind:
  - the mock system-admin runtime path is removed from the current tree
  - several major features now run through decomposed shells/services instead of only through the original hotspot files
- The biggest still-open traceability gaps are not missing features; they are incomplete cutover and normalization in the runtime/proof state layers.

## Findings
### Coverage result
- Every major user-visible feature has now been traced from frontend entry to backend surface and at least one data source.
- Proof and admin features have the deepest cross-layer coupling and therefore the highest risk concentration.
- Legacy-local and engineering-only verification surfaces are now clearly separated from the fully live product runtime.

### Blind spots remaining
- Curriculum linkage has weaker direct test coverage than core proof, session, or admin hierarchy features.
- Accessibility remains cross-cutting rather than a standalone feature, so it is represented through linked issue IDs rather than a dedicated feature row.

## Implications
- This matrix can now serve as the master feature map for engineering planning and regression scoping.
- Feature risk is no longer abstract; it is tied to exact files, endpoints, tables, and tests.
- Leadership can now distinguish live product breadth from transitional or dormant surfaces.

## Recommendations
- Use this matrix as the entrypoint for planning any refactor that touches shared hotspots such as `src/App.tsx`, `src/system-admin-live-app.tsx`, `academic.ts`, or `msruas-proof-control-plane.ts`.
- When a new feature is added, update this matrix before considering the audit pack current.
- Use the linked issue IDs here as the bridge between feature impact and remediation sequencing.

## Confirmed facts vs inference
### Confirmed facts
- Frontend entry points, backend endpoints, core files, database tables, and tests listed above are present in the codebase.
- Lifecycle status labels are grounded in whether the feature is active live runtime, mock-only, or transitional in current code.

### Strongly supported inference
- Some feature boundaries exist more clearly at the product level than at the code level. In the implementation, several “features” are still entangled inside giant orchestrator files.
